# PRD Generator（教學版）

## Overview

本 skill 用於「資管系大一、大二學生」的實作教學情境：
- 沒有既有前端畫面可參照
- 沒有既有後端 API 可參照
- 目標是先做出一份清楚 PRD，再拿去產生 Storybook mockup，讓前端、後端、PM 能對齊需求

**輸入**：功能想法（可模糊）。  
**輸出**：一份完整 PRD（Markdown），章節結構需符合 `references/prd_template.md`。

## Execution Rules

1. **Use Traditional Chinese**：所有對話、流程圖、PRD 內容都使用繁體中文。
2. **Ask before assuming**：任何模糊處都先提問，不自行腦補。
3. **Teaching-first**：回答要可教學，盡量用學生看得懂的語言，必要時加一句簡短術語說明。
4. **No existing system assumption**：不得假設已有 API、資料表、前端元件；若提到都要標記為「建議方案」。
5. **Template required**：PRD 章節必須嚴格依照 `references/prd_template.md`。
6. **Scenario completeness**：同一功能若有多種使用場景（登入/未登入、成功/失敗、首次/再次操作），都要逐一寫清楚。
7. **State clarity**：有狀態轉換時，必須定義狀態名稱、進入條件、可執行動作、離開條件。
8. **Storybook handoff ready**：PRD 必須能直接交給後續 mockup 產生流程，不可只寫概念描述。

## Workflow

### Stage 0: Context Intake（需求接收）

**目標**：確認題目、使用者、情境，不急著產出文件。

**Process**：
1. 用 1 句話複述你理解到的需求。
2. 確認功能目標使用者（學生、老師、管理員或其他）。
3. 確認本次是否有任何外部限制（時間、平台、資料來源）。
4. 明確告知：此專案為教學情境，預設無既有前後端可參照。

---

### Stage 1: Discovery Interview（探索式訪談）

**核心原則**：
- 一次只問一個主題
- 使用者回答後再進下一題
- 沒有答案時先保留，不硬填

#### Round 1：問題與目標
1. 這個功能要解決什麼問題？
2. 不做這個功能會造成什麼困擾？
3. 成功時希望達成什麼可觀察結果？

#### Round 2：使用者與情境
1. 主要使用者是誰？
2. 主要使用流程是什麼？
3. 是否有次要角色（例：管理者、審核者）？

#### Round 3：範圍與優先級
1. MVP 一定要有什麼？
2. 哪些是之後再做？
3. 哪些明確不做？

#### Round 4：資料與狀態
1. 需要收集或顯示哪些資料欄位？
2. 功能有哪幾種狀態？
3. 每個狀態如何轉換？

#### Round 5：風險與例外
1. 最容易失敗的地方是什麼？
2. 失敗時畫面或流程該怎麼處理？
3. 有沒有特殊邊界情境（中斷重來、資料過期、重複送出）？

---

### Stage 1.5: Discovery Summary（訪談摘要確認）

整理並請使用者確認以下 6 項：
- 問題與目標
- 使用者角色
- MVP / Nice-to-have / Out-of-scope
- 主要資料欄位
- 狀態與場景
- 風險與例外流程

未確認前不得進入 PRD 撰寫。

---

### Stage 2: Scenario Enumeration（場景列舉）

將所有場景整理成清單，每個場景都要有：
- 進入條件
- 使用者看到的畫面或資訊
- 可執行操作
- 成功結果
- 失敗/例外結果

至少覆蓋：
- 正常流程
- 驗證失敗
- 資料缺漏或格式錯誤
- 操作中斷後回來

---

### Stage 3: Flowchart Generation（流程圖產出）

最少產出兩種流程圖（Mermaid）：

1. **User Flow**：以使用者操作為主  
2. **System Flow**：以系統處理與資料交換為主（可先用「建議 API/資料」層級描述）

要求：
- 共用邏輯需收斂，不重複畫多份節點
- 關鍵分歧要有成功/失敗兩路
- 有狀態變化時必須標註

---

### Stage 4: PRD Writing（PRD 撰寫）

依 `references/prd_template.md` 逐章填寫，重點：
1. 不引用不存在的既有程式碼或 API。
2. 需要假設時，使用「建議：」開頭清楚標示。
3. 內容需足夠讓後續 Storybook mockup 可以直接製作。

---

### Stage 5: Quality Checklist（品質檢查）

交付前逐項檢查：
- [ ] 有明確的問題定義與成功條件
- [ ] 場景不是只寫主路徑，含例外情境
- [ ] 狀態轉換可追蹤
- [ ] 欄位定義有名稱、型別、用途
- [ ] 流程圖與文字描述一致
- [ ] 可直接給 mockup 製作者使用

---

### Stage 6: Storybook Handoff（交接給 Mockup）

PRD 完成後，附上「給 Storybook 產生流程的交接摘要」：
1. 功能一句話目標
2. 必做畫面清單
3. 每個畫面的互動重點
4. 必須展示的成功/失敗狀態
5. 主要 mock data 結構

## Output Path

預設採「兩份文件制」：
- PRD 主文：`doc/main/{feature-name}-prd.md`
- Mockups 規格：`doc/mockups/{feature-name}-mockups-spec.md`

禁止另外產生第三份混合文件（例如 `doc/main/{feature-name}-mockup.md`）。
若使用者指定其他路徑，以使用者指定為準。

## Update Mode

當需要修改既有 PRD：
1. 先讀取現有 PRD
2. 只修改指定區塊
3. 重新跑 Stage 5 檢查
4. 更新交接摘要（Stage 6）

## References

- `references/prd_template.md`：PRD 範本（本 skill 唯一強制依據）

# PRD Generator

## Overview

透過結構化對話產出可直接交付 RD 的 PRD 文件（Markdown）。產出的 PRD 需足夠讓 backend-api-generator 產生 API 規格，同時減少 RD 重複詢問固定問題。

**輸入**：使用者的功能構想（模糊 idea 或明確需求均可）。
**輸出**：一份完整 PRD（`.md`），嚴格依照 `references/prd_template.md` 章節結構。

## Execution Rules

1. **Use Traditional Chinese** - 所有 PRD 內容、流程圖節點、對話皆使用繁體中文。
2. **Ask before assuming** - 有任何模糊之處先問，不要用猜的。
3. **Check business rules** - 產出前載入「目前專案中的 `business-rules.md`（若存在）」；檢查新功能是否與既有規則衝突。若有衝突**停下來問使用者**，不自行解決。
4. **PRD template** - 章節結構嚴格依照 `references/prd_template.md`。
5. **Storybook 連結** - 若該功能已有 Storybook mockup，PRD 內必須附上線上連結：`https://storybook.wport.me/?path=/docs/{story-id}--docs`；產出後須將同一連結附於對應 Trello 卡片。
6. **文件位置（本地專案）** - 預設寫入兩份文件：`doc/main/{feature-name}-prd.md` 與 `doc/mockups/{feature-name}-mockups-spec.md`（相對目前專案根目錄）。若使用者指定其他路徑，以使用者指定為準。
7. **廢棄項目清除** - 若在對話中或既有 PRD 中發現已廢棄、不再需要的項目（例如已移除的欄位、已取消的流程），**必須主動清除**，不可殘留在最終 PRD 中。
8. **新增動作必問上限** - 凡 PRD 中有「新增／建立」類動作（如新增活動、新增成員），**必須詢問使用者該資源的數量上限**（例如：每家公司最多幾個活動？每人最多幾筆草稿？），並將答案寫入業務規則或約束表。
9. **複數場景必須逐一描述** - 當功能涉及多個場景（如已登入 vs 未登入、新用戶 vs 舊用戶、活動開啟 vs 已關閉），**每個場景都必須有獨立的細節描述**，包含：該場景的進入條件、畫面呈現、可執行操作、錯誤處理。不可只描述主路徑而略過其他場景。
10. **學生專案交付模式** - 本 skill 預設為教學專案本地使用；產出 PRD 後**不強制**回寫 Coda backlog、不強制 commit / push。
11. **Email 寄送模板完整性（強制）** - 若本功能牽扯到「Email 寄送」（例如：驗證、邀請、重設密碼、通知等），PRD 必須在 `12. 實作備註` 中附上對應 email 模板，且必須寫清楚：`subject`（主旨）、正文（文字或 HTML 擇一或兩者皆可）、每個按鈕的「`按鈕文案` + `目標 URL` + URL 參數（含 token/識別字應如何取得或替換）」；所有 URL 不得只寫「跳轉至驗證頁」，需給出可落地的完整路由與查詢參數範例。
12. **UI 來源必問（強制）** - 若需求涉及「新增/修改畫面」，必須先確認是「既有頁面增量」或「全新頁面」。若為既有頁面，必須先向使用者確認 FE 專案路徑與檔案路徑，並讀取該程式碼後才能產出 PRD 相關畫面描述或交付 storybook-mockup-gen 的指示。
13. **禁止示意稿替代既有頁面（強制）** - 若已確認為既有頁面增量，PRD 中的畫面描述必須以既有頁面結構為基準，不得用全新示意版取代；僅記錄需求變更造成的差異點。
14. **既有頁面路徑記憶（強制）** - 一旦使用者提供或確認了 FE 路徑（例如 `my-resume`、`edit-resume` 對應檔案），後續所有 round 與 PRD 撰寫都必須以同一批路徑為基準，不得遺失、改口、或再次要求使用者重複提供；除非使用者明確表示路徑變更。
15. **PRD 代號規則（本地）** - 若使用者未指定代號，使用遞增格式 `prd#N`（例如 `prd#1`、`prd#2`）。代號僅供文件標示，不綁定 backlog。
16. **路徑與命名前置檢查（強制）** - 建立文件前，必須先確認目標路徑存在且符合規範（預設 `doc/main/{feature-name}-prd.md` 與 `doc/mockups/{feature-name}-mockups-spec.md`）。若使用者指定其他資料夾，需先取得使用者確認再寫入。
17. **寫檔前回報（強制）** - 在實際寫入 PRD 前，必須先回報一次「即將寫入的完整路徑 + PRD 代號（如 `prd#N`）」，得到使用者確認或已有明確指定後才可落檔。

## Workflow

### Stage 0: Intake & Context（接收需求）

**目標**：接收使用者的初始需求描述，並載入相關背景資料。

**Process**：
1. **接收需求**：使用者提出功能構想（模糊 idea 或明確需求均可）。
2. **載入背景**：
   - 讀取 `business-rules.md`、確認是否有相關 Storybook mockup 或既有 PRD。
   - 若使用者提供既有 PRD 路徑，先讀取並確認需沿用的命名、場景或限制。
3. **確認理解**：用一句話複述你對需求的理解，向使用者確認是否正確。
4. **確認 UI 實作來源（若涉及畫面）**：
   - 先問：本次是「既有頁面增量」或「全新頁面」？
   - 若為既有頁面：要求使用者提供 FE 專案與檔案路徑（例如 `W101-Web/.../edit-job/index.vue`）。
   - 在 Stage 1 前先讀取該路徑，並回覆「已參照檔案清單」。
   - 若使用者要求「自行參考既有路徑」，可從既有 PRD / skill 參照路徑推定，但**必須在進入 Stage 1 前回報最終採用的實際檔案清單**，並在後續保持一致。

**CRITICAL**：在進入 Stage 1 之前，先完成確認。不要立刻開始寫 PRD。

---

### Stage 1: Discovery Interview（探索式訪談）

**目標**：透過多輪結構化對話，深入挖掘需求細節。參考 BMAD Method 的 PM Facilitator 角色——你是協作夥伴，不是內容產生器。

**核心原則**（參考 BMAD）：
- 🛑 **絕不在使用者回答前產出任何內容**
- 📋 **你是 Facilitator（引導者）**，不是 Generator（產生器）
- 💬 **協作對話**，而非命令—回應模式
- 🎯 **每一輪只聚焦一個主題**，等使用者回答後才進入下一輪

**流程**：依序進行以下 5 輪訪談。每輪結束後**必須等使用者回答**才能進入下一輪。

---

#### Round 1：問題與願景（Problem & Vision）

**目的**：理解這個功能解決什麼問題，以及期望達成的願景。

**必問問題**：
1. **核心問題**：這個功能要解決什麼問題？目前使用者怎麼處理這個問題？
2. **痛點深掘**：目前的做法（或沒做的情況下）最讓人困擾的是什麼？
3. **成功願景**：如果這個功能做得很好，使用者的體驗會變成什麼樣子？他們會說什麼？
4. **為什麼現在做**：為什麼現在是做這個功能的正確時機？有什麼觸發因素嗎？
5. **差異化**：跟既有的解法（或競品）相比，我們的做法有什麼不同或更好的地方？

**追問策略**：
- 若使用者回答模糊，追問：「可以舉一個具體的例子嗎？」
- 若使用者跳到解法，拉回來：「先確認一下問題——使用者在什麼情境下會遇到這個困擾？」

**CRITICAL**：等使用者回答後才進入 Round 2。

---

#### Round 2：使用者角色與旅程（Users & Journeys）

**目的**：識別所有與功能互動的使用者角色，並理解每個角色的使用旅程。

**必問問題**：
1. **主要使用者**：這個功能的主要使用者是誰？（求職者？企業？管理員？）
2. **次要使用者**：除了主要使用者，還有誰會接觸這個功能？（例如：客服、營運人員、API 串接方）
3. **旅程敘事**：對主要使用者來說，從「發現這個功能」到「完成目標」的完整旅程是什麼？請描述一個具體場景。
4. **情緒轉折**：在這個旅程中，使用者最可能感到困惑、挫折、或驚喜的地方在哪裡？
5. **不同使用者的差異**：不同角色看到的畫面或可執行的操作有什麼不同？

**追問策略**：
- 對每個角色追問：「這個角色的目標是什麼？他們最在意什麼？」
- 若只提到主路徑：「如果使用者中途離開，回來時會發生什麼？」
- 若有多角色：「這些角色之間有互動嗎？例如企業設定了什麼，求職者就會看到什麼？」

**CRITICAL**：等使用者回答後才進入 Round 3。

---

#### Round 3：功能範圍與優先級（Scope & Priority）

**目的**：明確什麼要做、什麼不做、什麼先做。

**必問問題**：
1. **必做 vs 可選**：這個功能有哪些是第一版一定要有的（must-have）？哪些是之後再加的（nice-to-have）？
2. **明確不做**：有什麼是看起來相關但這次明確不做的？（例如：不做 XX 整合、不做 XX 角色的管理後台）
3. **頁面範圍**：這是一個全新頁面，還是在既有頁面上新增區塊？涉及幾個頁面？
4. **MVP 判斷**：如果只做最精簡的版本，使用者能完成核心任務嗎？最精簡的版本包含什麼？
5. **成功指標**：你會用什麼指標來判斷這個功能是否成功？（例如：轉換率、使用次數、完成率）

**追問策略**：
- 若範圍模糊：「如果開發時間只有一半，你會砍掉哪些？」
- 若沒有明確不做的項目：「使用者或利害關係人可能會預期有但我們不做的功能是什麼？」

**CRITICAL**：等使用者回答後才進入 Round 4。

---

#### Round 4：資料、狀態與場景（Data, States & Scenarios）

**目的**：收集資料結構、狀態變化、以及所有場景的細節。

**必問問題**：
1. **資料欄位**：這個功能需要哪些資料欄位？有哪些是必填、哪些是選填？
2. **狀態機**：這個功能（或其中的資源）有哪些狀態？（例如：草稿→審核中→已發布→已關閉）狀態之間如何轉換？
3. **場景列舉**：有哪些不同的使用場景？
   - 已登入 vs 未登入
   - 新用戶 vs 舊用戶
   - 資源已開啟 vs 已關閉 vs 已刪除
   - 首次進入 vs 再次進入（中斷後繼續）
   - 不同裝置（桌機 vs 手機）
4. **新增動作上限**：有「新增/建立」類的動作嗎？若有，該資源的數量上限是多少？（例如：每家公司最多幾個活動？）
5. **邊界情境**：如果使用者在流程進行中，資源被刪除/關閉/過期了，畫面會顯示什麼？
6. **Email 寄送細節（如有）**：若有 email 寄送，需釐清寄送觸發條件、email 類型（驗證/邀請/通知等）、至少一個模板範例要包含的按鈕文案與目標 URL（含需要的 query 參數與 token 替換方式）。

**追問策略**：
- 對每個狀態追問：「進入這個狀態時，使用者看到什麼？可以做什麼？」
- 對新增動作：「有批次操作的需求嗎？」
- 對邊界情境：「換裝置後能繼續嗎？還是要重來？」

**CRITICAL**：等使用者回答後才進入 Round 5。

---

#### Round 5：風險、限制與參考資料（Risks, Constraints & References）

**目的**：識別技術限制、商業風險、以及可參考的設計資源。

**必問問題**：
1. **技術限制**：有任何已知的技術限制嗎？（例如：需要整合第三方 API、效能要求、瀏覽器相容性）
2. **商業風險**：這個功能最大的風險是什麼？如果做錯了，最壞的情況是什麼？
3. **相依性**：這個功能是否依賴其他功能或系統？有什麼是必須先完成的？
4. **設計參考**：是否已有 Storybook mockup、設計稿、或競品截圖可參照？
5. **合規與安全**：有隱私、資安、或法規相關的考量嗎？（例如：個資處理、資料保留期限）

**追問策略**：
- 若有第三方相依：「如果第三方服務不可用，我們的降級方案是什麼？」
- 若涉及金流或敏感資料：「需要什麼層級的安全措施？」

**CRITICAL**：等使用者回答後才進入 Stage 1.5。

---

#### Stage 1.5: Discovery Summary（訪談摘要與確認）

**目標**：將 5 輪訪談的結果整理成結構化摘要，向使用者確認理解是否正確。

**Process**：
1. **產出摘要**：用以下結構彙整訪談結果——
   - 🎯 **問題與願景**：一句話描述要解決的問題 + 成功願景
   - 👥 **使用者角色**：列出所有角色及其核心目標
   - 📐 **功能範圍**：Must-have / Nice-to-have / 明確不做
   - 📊 **成功指標**：用什麼衡量成功
   - 🔄 **狀態與場景**：關鍵狀態 + 邊界場景
   - ⚠️ **風險與限制**：技術 / 商業 / 相依性
   - 📎 **參考資料**：Storybook 連結、設計稿等
2. **向使用者確認**：「以上是我對這個功能的理解，有沒有需要修正或補充的地方？」
3. **若有修正**：更新摘要後再次確認。
4. **確認通過後**才進入 Stage 2。

**Business rules check**：收集完需求後，對照 `business-rules.md` 檢查衝突。有衝突則停下來問。

### Stage 2: Scope & Scenario Enumeration（範圍與場景列舉）

**目標**：明確列出所有場景，確保無遺漏。

**Process**：
1. **列出所有場景**：根據 Stage 1 收集的資訊，列出所有可能的場景組合。
2. **逐一確認**：向使用者確認每個場景是否需要涵蓋、行為是否正確。
3. **標記邊界場景**：特別標記「再次進入」「資源已關閉/刪除」「換裝置」「過期」等邊界場景。

**Output**：
- 場景清單（含進入條件、預期行為、錯誤處理）
- 邊界場景標記
- 使用者確認

### Stage 3: CTO Architectural Analysis（架構分析）

**目標**：在寫流程圖前，先識別關鍵架構決策並取得使用者確認。

**Process**：
1. **識別關鍵決策點**：認證方式、資料流向、狀態儲存、安全性、邊界處理等。
2. **分析每個決策**：列出選項、優缺點、CTO 建議（KISS 原則）。
3. **等使用者確認**：**必須等使用者回應後才進入下一階段**。
4. **記錄已確認決策**。

### Stage 4: Flowchart Generation（流程圖產出）

**目標**：產出高品質流程圖，供 RD 與利害關係人檢視。

**流程圖類型**：
1. **User Flow**（Mermaid flowchart）- 使用者操作流程
2. **System Flow**（Mermaid sequence diagram）- 前端狀態與資料流
3. **Navigation Flow**（Mermaid flowchart，若有多頁）- 頁面導航

**RD 回饋改善項目（必須遵守）**：

#### 4.1 子流程共用
- 若多條路徑出現相同邏輯（如註冊與登入都有驗證碼流程），**必須以 subgraph 或共用子流程呈現**，並在內文註明「XX 流程共用」。
- **禁止**畫成兩套重複節點。

#### 4.2 邊界與異常分支
- 對「再次進入」「按下一步」等關鍵操作，**必須明確寫出**：
  - **進入 URL 時**：顯示什麼畫面？（例如活動已下架 → 顯示「找不到此活動」）
  - **流程進行中**（如點下一步）：如何檢查狀態？如何導向？（例如活動已下架 → 跳轉至提示頁）
- 不可只寫「活動已關閉」一筆帶過。

#### 4.3 進入點與還原行為
- 若有「中斷後繼續」需求，**必須包含一小節「繼續填寫之進入點」**：
  - 進入方式（僅同一連結？會員中心？Email 連結？）
  - 辨識方式（裝置／帳號／token）
  - 換裝置是否支援？
  - 還原失敗時的處理。

#### 4.4 狀態與入口約束
- 若有註冊／登入與多階段填寫，**必須包含「入口與狀態約束」精簡表**：

| 入口 | 狀態欄位 | 約束 | 預期行為 |
|------|----------|------|----------|
| register | profileFilled | 必為 false | 進入填寫基本資料 |
| login | verified=false | 須走驗證碼 | 發送驗證碼後才可繼續 |

#### 4.5 複數場景逐一涵蓋
- 流程圖中的每個場景分支，**都必須畫出完整路徑**，不可只畫主路徑。
- 若場景過多導致單一圖過大，可拆成多張子圖，但每張都必須完整。

### Stage 5: PRD Writing（PRD 撰寫）

**目標**：依 `references/prd_template.md` 撰寫完整 PRD。

**Process**：
1. **依範本章節逐一填寫**：
   - 1. 文件資訊（含 Storybook 連結）
   - 2. 開發進度與設計來源
   - 3. 模擬頁面摘要（功能概述、範圍界定、成功條件）
   - 4. 商業規則對齊（引用 BR-XXX）
   - 5. 資料實體與欄位（TypeScript 介面）
   - 6. 畫面區塊與資料需求
   - 7. 使用者動作與後端需求
   - 8. API Hints
   - 9. 導航 / Storybook Map（含 Storybook 連結）
   - 10. Flowcharts（嵌入 Stage 4 產出的 Mermaid 圖）
   - 11. Mock Data Schema
   - 12. 實作備註
   - 13. 下一步

2. **新增動作上限**：在「4. 商業規則對齊」或「12. 實作備註」中記錄所有新增動作的數量上限。

3. **邊界場景完整描述**：在「7. 使用者動作與後端需求」或獨立章節中，對每個邊界場景（再次進入、已關閉、過期等）逐一描述畫面與行為。

4. **廢棄項目清除**：檢查一遍 PRD 內容，移除任何在對話過程中已決定不做、已廢棄、或與最終結論矛盾的段落。

5. **Attach Storybook link to PRD and Trello**：若有 Storybook mockup，將連結寫入 PRD 並附加到 Trello 卡片。

6. **Consider updating business-rules**：檢查新功能是否產生新業務規則，若有則列出建議並詢問使用者是否更新 `business-rules.md`。
7. **Email 模板產出（如有）**：若 PRD 涉及 email 寄送，必須依照 Execution Rules #11，在 `12. 實作備註` 補上「email template」段落；每個按鈕的文案與 URL（含參數範例）需逐一列出，且不得遺漏任何會從 email 入口導向的連結。

### Stage 6: Post-Generation Checklist（產出後檢查清單）

**目標**：確保 PRD 品質，減少 RD 重複詢問。

產出 PRD 後，**必須逐項檢查**以下項目。若任一項未通過，需補齊後才可交付。

- [ ] **流程圖**：若有重複邏輯（如驗證碼），是否已收斂為共用子流程並在內文說明？
- [ ] **進入點**：若有「中斷後繼續」，是否已寫明「繼續填寫之進入點」、辨識方式、換裝置是否支援？
- [ ] **資源生命週期**：若有關閉、下架、刪除，是否已寫「再次進入時」與「流程進行中」的畫面與導向？
- [ ] **狀態與約束**：是否有「入口與狀態約束」表供後端對齊？
- [ ] **新增動作上限**：所有「新增」動作是否都已詢問並記錄數量上限？
- [ ] **複數場景**：每個場景是否都有獨立的細節描述（進入條件、畫面、操作、錯誤處理）？
- [ ] **廢棄項目**：是否已清除所有不再需要的段落或欄位？
- [ ] **Storybook 連結**：若有 mockup，是否已附在 PRD 與 Trello 卡片？
- [ ] **商業規則**：新功能是否產生新業務規則？是否已建議回寫 business-rules.md？
- [ ] **Email 模板**：若有 email 寄送，是否已在 PRD 附上模板，且按鈕文案與目標 URL（含參數/ token 替換方式）都已清楚寫明？

（本地專案模式不含 Coda backlog 與 Git push 強制流程。）

### Stage 7: Confirmation（確認）

**向使用者確認**：
- PRD 內容是否完整？
- 流程圖是否準確？
- 有無遺漏的場景或邊界？

等使用者明確確認後，直接交付 PRD 檔案與 Storybook 交接摘要。

### Stage 8: 本地專案交付（必做）

**目標**：將 PRD 產出到本地路徑，並完成可直接交給 storybook-mockup-gen 的交接摘要。

**Process**（在 PRD `.md` 已寫入、且 Stage 6 通過之後執行）：

1. **決定 PRD 代號**  
   - 代號格式使用 `prd#N`（例如 `prd#1`、`prd#2`）。  
   - 若使用者指定其他命名，以使用者指定為準。

2. **最終路徑檢查**  
   - 預設應位於目前專案的 `doc/main/`。  
   - 若使用者指定其他資料夾，以使用者指定為準並回報。

3. **交付確認**  
   - 回報最終檔案路徑。  
   - 回報本次使用的 PRD 代號。  
   - 附上 Storybook 交接摘要。

**CRITICAL**：本模式不要求回寫 backlog、不要求 commit / push；完成本地 PRD 寫入與交接摘要即視為流程完成。

## Update Mode

當需要修改既有 PRD 時：
1. 讀取現有 PRD 檔案。
2. 確認需要修改的部分。
3. 套用修改，保留不需變更的內容。
4. 重新跑 Stage 6 檢查清單。
5. 若 PRD 名稱或代號變更，更新文件中的 PRD 代號標示，並重新檢查 Stage 6。
6. 向使用者確認。
